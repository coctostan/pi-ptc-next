<objective>
## Goal
Codify the `details.ptcValue` normalization contract for `code_execution` so structured tool results are preferred as the machine-native value, while existing text normalization remains intact for tools that do not provide `ptcValue`.

## Purpose
Milestone 1 already aligned nested tool execution with the same active Pi tools used in chat. This phase locks down the next trust layer: when those active tools return structured payloads, Python should receive them unchanged instead of reparsing chat-oriented text.

## Output
- Any minimal runtime adjustment needed in `src/tool-adapters.ts` to make `ptcValue` preference explicit and stable
- Targeted adapter tests for mocked hashline-style `read` / `grep` / `edit` structured payloads
- A narrow RPC-level regression test proving structured values pass through the nested tool boundary unchanged
</objective>

<acceptance_criteria>
## AC-1: Structured machine values are preferred when present
```gherkin
Given a nested tool result includes `details.ptcValue`
When `normalizeToolResult()` processes that result
Then it returns the `ptcValue` unchanged as the normalized machine value instead of deriving a value from text content
```

## AC-2: Existing text fallback behavior stays intact
```gherkin
Given a nested tool result does not include `details.ptcValue`
When `normalizeToolResult()` processes built-in and custom tool outputs
Then existing text-based normalization behavior remains unchanged for `read`, `grep`, and generic custom-tool results
```

## AC-3: Structured values cross the RPC tool boundary unchanged
```gherkin
Given the nested RPC protocol receives a tool result whose `details.ptcValue` contains mocked hashline-style structured data
When `RpcProtocol` emits the `tool_result` frame back to Python
Then the frame's `value` matches the structured payload exactly and existing text-only protocol behavior still passes
```
</acceptance_criteria>
