<objective>
## Goal
Stabilize the `code_execution` Python-source payload contract so running updates, successful completions, and failures all carry source through structured details/metadata without adding source to normal result text.

## Purpose
Milestone 21 depends on source being consistently available before Phase 64 changes TUI presentation. Phase 62 found that progress updates already carry `details.userCode`, successful completions drop it, generic partial updates omit it, and failures throw without source-bearing details.

## Output
A TDD implementation that preserves the existing `details.userCode` field and adds source-bearing error details for `code_execution` failures, with focused tests proving success, partial, and failure payload behavior.
</objective>

<acceptance_criteria>
## AC-1: Success Details Include Source
```gherkin
Given a `code_execution` invocation completes successfully
When `RpcProtocol` resolves the completion result and `buildCodeExecutionTool.execute` returns the tool result
Then `details.userCode` contains the original user Python source lines and normal text content contains only execution output, not an added source dump
```

## AC-2: User-Code Failure Results Include Source
```gherkin
Given a `code_execution` invocation fails through framed Python error or pre-terminal Python failure
When `buildCodeExecutionTool.execute` handles the user-code Python failure
Then it returns a structured failed result whose details include `userCode`, failure/error metadata, traceback/raw error data where applicable, and telemetry/recovery data, without appending Python source to normal content or the human error message
```

## AC-3: Partial Updates Deliberately Carry Source
```gherkin
Given `RpcProtocol` emits progress, update, or nested tool-call partial updates during execution
When the extension forwards partial result details to Pi
Then each partial details payload includes `userCode` alongside existing metrics and any current-line metadata, preserving existing progress behavior
```

## AC-4: Report and Existing Error Semantics Stay Compatible
```gherkin
Given existing report-producing successes and typed execution errors
When Phase 63 source details are added
Then `details.report` / `details.reportProduced`, error class names, raw Python message formatting, timeout/abort/transport/protocol behavior, and existing focused tests remain compatible
```
</acceptance_criteria>
