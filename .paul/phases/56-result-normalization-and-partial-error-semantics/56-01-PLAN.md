<objective>
## Goal
Tighten three live-audit pain points (issues 6, 7, 10 from `docs/issues/2026-05-13-code-execution-helper-edge-cases.md`) by:

1. **Path normalization parity** — apply the same workspace-relative path policy to direct `read()` results, `ptc.read_text()`, `ptc.read_many()`, and any `read`/`grep` payload surfaced through `ptc.batch_tool(...)` (already applied to direct `grep`).
2. **`ptc.read_many` partial-error envelope** — keep returning `list[str]` by default but expose a typed partial signal (opt-in `on_error="collect"` returning a structured `kind: "read_many_partial"` envelope) so missing files surface as failures, not as raw error strings concatenated into otherwise-successful contents.
3. **`ptc.batch_tool(..., on_error="collect")` tool-level failure classification** — detect normalized tool-error payloads (`{ok: false, error: ...}` or `kind: "execution_error"`) inside successful transport calls and reflect them in `stats.failed` and per-entry `ok` while preserving raw `value` for callers.

## Purpose
These are the remaining Milestone 19 runtime-correctness gaps surfaced by live manual testing. They directly threaten the core value of `pi-ptc-next` (Python helpers behaving the way prompt/docs say they do): silent tool-level failures and inconsistent path shapes mislead aggregation and provenance code running inside `code_execution`.

## Output
- Updated `src/python-runtime/runtime.py` with normalized read/grep result paths, an opt-in `read_many` partial envelope, and tool-level failure detection in `batch_tool` collect mode.
- Generated `code_execution` tool description (in `src/index.ts`) and README guidance refreshed to describe the new envelopes and the documented invariants.
- New focused regression tests proving each behavior; existing live-audit + orchestration tests still pass.
- `.paul/phases/56-result-normalization-and-partial-error-semantics/56-01-SUMMARY.md` (produced during UNIFY).
</objective>

<acceptance_criteria>

## AC-1: Direct read + helper read paths normalize consistently
```gherkin
Given the user invokes either `await read(path="README.md")`, `ptc.read_text("README.md")`, or `ptc.read_many(["README.md"])` from inside `code_execution`
And the file is under the host workspace root
When the structured result is inspected
Then any `path` field present in the result (including ReadResult `path`/`lines[].path` and per-entry helper metadata) is normalized to a workspace-relative path (e.g. `README.md`), matching the existing `_normalize_grep_result` policy
And paths outside the workspace root remain absolute
```

## AC-2: `ptc.batch_tool` propagates the same path-normalization policy
```gherkin
Given a `ptc.batch_tool([...])` call that includes `read` or `grep` entries returning structured results
When `on_error="raise"` (default) or `on_error="collect"` modes complete
Then each per-entry `value` (collect mode) or list item (raise mode) carries the same workspace-relative path shape that the direct wrapper would produce
And no other field in the orchestration envelope changes
```

## AC-3: `ptc.read_many` exposes a typed partial envelope
```gherkin
Given `ptc.read_many(paths)` is called with at least one missing or unreadable path
When the caller passes `on_error="collect"`
Then the helper returns a structured envelope of shape `{"kind": "read_many_partial", "results": [...], "stats": {"total", "succeeded", "failed"}}`
And each result entry is `{"index", "path", "ok": True, "value": str}` for success or `{"index", "path", "ok": False, "error": str}` for failure
And calling without `on_error` (default) preserves the existing `list[str]` contract for backwards compatibility, but emits a structured `error` string for the failed entry rather than embedding a stack-like payload
```

## AC-4: `ptc.batch_tool(..., on_error="collect")` classifies tool-level failures as failed
```gherkin
Given a `ptc.batch_tool([...], on_error="collect")` call whose underlying tool returns a normalized error payload (e.g. `{"ok": false, "error": ...}` or a normalized result with `kind: "execution_error"`)
When the batch completes without raising a Python/transport exception
Then the per-entry record has `ok: False`, retains the original payload under `value`, and includes a string `error` summary
And `stats.failed` reflects this entry while `stats.succeeded` does not
And transport/Python exceptions continue to be classified as failed (unchanged behavior)
```

## AC-5: Generated guidance and README describe the new contracts
```gherkin
Given a fresh `code_execution` tool description and `README.md` after this phase
When a reader scans the helper docs
Then `ptc.read_many` documents both the default `list[str]` form and the opt-in `on_error="collect"` envelope (with `kind: "read_many_partial"`)
And `ptc.batch_tool` documents that `on_error="collect"` now flags tool-level error payloads as failed entries
And the path-normalization invariant is stated once near the read/grep helper section so callers know wrapper, helper, and batched paths agree
```
