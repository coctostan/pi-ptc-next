<objective>
## Goal
Make completed `code_execution` results debuggable by preserving the Python source behind Pi's existing tool-result expansion affordance, and add a model-facing routing clarification for `nu` versus `code_execution`.

## Purpose
Milestone 18's thesis is that PTC should convert "I need to know X about this repo / dataset" into a small, well-shaped answer with minimum agent friction. Phase 49 improves the user-facing debugging loop before the structured report work starts: users should be able to inspect what Python the agent ran after the tool finishes, without forcing long code blocks into the default collapsed view. The phase also records the Pi TUI primitive audit that Phase 50 will reuse for report rendering.

## Output
- `.paul/phases/49-pi-tui-audit-and-collapsible-code-body/49-01-PI-TUI-AUDIT.md` documenting TUI findings and chosen implementation primitive.
- `src/index.ts` render-path update: completed `code_execution` results support collapsed and expanded views using `details.userCode`.
- Prompt metadata update clarifying when to prefer `nu` vs `code_execution`.
- Focused render/prompt tests plus README and CHANGELOG notes.
</objective>

## AC-1: Completed code_execution results preserve source without default noise
```gherkin

And the live partial execution view with the current-line cursor remains unchanged
```
