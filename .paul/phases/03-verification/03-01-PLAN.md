<objective>
## Goal
Document and surface the structured-results contract that Phase 2 made real, so `code_execution` users and maintainers clearly understand when active tools can return richer `details.ptcValue` payloads for `read` / `grep` / `edit`, while text fallback behavior remains valid.

## Purpose
The runtime now preserves hashline-style structured values end to end, but the current repo guidance still reads like the default helper signatures are the whole story. This phase closes that trust gap with focused docs and model-facing guidance before the later real two-extension smoke proof.

## Output
- Updated repo docs that describe representative structured payload shapes and fallback behavior for hashline-native `read` / `grep` / `edit`
- Updated `code_execution` tool description that warns models about richer override payloads arriving through `details.ptcValue`
- Focused regression coverage proving the new guidance is surfaced when the extension boots
</objective>
