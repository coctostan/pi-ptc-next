  <action>
    Create `test/live-audit-helpers.test.ts` using Node's built-in test runner. This file exercises every Python helper with real tool calls against the live repo.

    Structure the test as a single describe block with sub-describes per category:

    **Core wrappers (5 tests):**
    - `read("package.json")` — assert returns content containing "name"
    - `grep("version", path=".", glob="package.json")` — assert returns matches
    - `find("*.ts", path="src")` — assert returns file paths
    - `ls(path="src")` — assert returns directory entries
    - `glob("src/**/*.ts")` — assert returns TypeScript file paths

    **Batch/file helpers (6 tests):**
    - `ptc.read_many(["package.json", "tsconfig.json"])` — assert returns list of 2 strings
    - `ptc.read_tree(pattern="**/*.ts", path="src", max_files=5)` — assert returns list of dicts with "content" keys
    - `ptc.find_files(pattern="*.ts", path="src")` — assert returns list of relative paths
    - `ptc.find_files_abs(pattern="*.ts", path="src")` — assert returns list of absolute paths
    - `ptc.read_text("package.json")` — assert returns a string containing "name"
    - `ptc.gather_limit([read("package.json"), read("tsconfig.json")], limit=2)` — assert returns list of 2

    **Orchestration helpers (3 tests):**
    - `ptc.batch_tool([{"tool": "read", "params": {"path": "package.json"}}, {"tool": "read", "params": {"path": "tsconfig.json"}}])` — assert returns list of 2 results
    - `ptc.first_success([{"tool": "read", "params": {"path": "NONEXISTENT.md"}}, {"tool": "read", "params": {"path": "package.json"}}])` — assert returns the package.json content (fallback)
    - `ptc.reduce_tool([{"tool": "read", "params": {"path": "package.json"}}, {"tool": "read", "params": {"path": "tsconfig.json"}}], "lambda acc, val: acc + '---' + str(len(str(val)))", "''")` — assert returns a string with separator

    **Output/assertion helpers (3 tests):**
    - `ptc.fit_output({"a": 1, "b": list(range(100))}, max_items=5)` — assert result is truncated
    - `ptc.expect_kind(read("package.json"), "string")` — assert does not throw (or returns the value)
    - `ptc.json_dump({"key": "value"})` — assert returns a JSON string

    **Introspection helpers (2 tests):**
    - `ptc.list_callable_tools()` — assert returns a list of dicts with "name" keys
    - `ptc.get_tool_schema("read")` — assert returns a dict with parameter info

    **Handle helpers (2 tests):**
    - `ptc.extract_handles(web_search_result_or_mock, kind="response")` — test with a real web_search call if available, or skip gracefully
    - `ptc.first_handle(result, kind="response")` — same approach

    Each test should use `code_execution` to run the Python code, capture the output, and assert on the result. Since this is a Node test file that can't directly invoke `code_execution` interactively, the tests should instead:

    1. Import the CodeExecutor and related types from the built dist
    2. OR — if direct executor invocation is impractical from test context — write the tests as **deterministic contract tests** that validate the Python runtime surface by importing `runtime.py` helper signatures and testing them through the existing test harness patterns used in `test/code-executor.test.ts`

    Use whichever approach the existing test infrastructure supports. Look at how `test/code-executor.test.ts` and `test/orchestration-ecosystem-helper.test.ts` test Python execution for the pattern to follow.

    The key requirement: each helper must be exercised through the actual Python runtime path (not mocked), and the result must be checkable.
