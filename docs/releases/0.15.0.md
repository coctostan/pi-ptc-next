# Release Notes — 0.15.0

`0.15.0` is the publishable-fork baseline where the package surface is renamed to `pi-ptc-advanced` while the repository lineage remains `pi-ptc-next`.

## What this release is for

This release makes the fork coherent for publication and maintenance without changing the project’s core runtime value.

The core value is unchanged:
- `code_execution` should invoke the same active Pi tool implementations the user sees in chat
- hashline-native `read` / `grep` / `edit` behavior should stay trustworthy inside Python workflows
- personal maintenance should be reproducible without relying on old session artifacts

## Fork lineage and naming

This baseline intentionally preserves three distinct identities:

- original upstream: [`cegersdoerfer/pi-ptc`](https://github.com/cegersdoerfer/pi-ptc)
- ongoing fork repository: [`coctostan/pi-ptc-next`](https://github.com/coctostan/pi-ptc-next)
- current package publish target: `pi-ptc-advanced`

The repository name did not change in this phase. Instead, the package/release surface was updated so users and maintainers can distinguish repository history from the package identity that would ship.

## Install / update target

Install directly from the fork repository:

```bash
pi install git:github.com/coctostan/pi-ptc-next
```

That remains the documented install path for this baseline. The package metadata inside the repo now targets `pi-ptc-advanced@0.15.0`.

If you already have the fork checked out locally, update your branch using your normal manual git workflow, then re-run the verification commands below.

## Verification commands

For the `0.15.0` baseline, the expected repo-local verification flow is:

```bash
npm run verify:personal
npm run verify:personal:full
npm run verify:ci
npm run verify:release-package
```

What they mean:
- `npm run verify:personal` — focused verification for routine local maintenance
- `npm run verify:personal:full` — higher-confidence full-suite verification
- `npm run verify:ci` — GitHub Actions parity verification, matching the repo-owned workflow command used by `.github/workflows/ci.yml`
- `npm run verify:release-package` — package metadata + `npm pack --dry-run` tarball verification plus a clean temporary install proof for the current publish surface

## Major shipped capabilities in this baseline

### Publish/package surface
- package metadata and release-package verification now target `pi-ptc-advanced@0.15.0`
- README and maintainer docs now describe the active publish baseline coherently
- fork lineage is explicit so the repo history and package target are not conflated

### Runtime / tool behavior retained
- `code_execution` still routes overridden builtin tools through the active Pi executors visible in chat
- structured `details.ptcValue` payloads are still preserved across the RPC boundary
- Python helper contracts and generated wrappers still expose structured anchored result models for builtin tooling
- callable-tool policy behavior remains explicit, including `ptc.callable`, `ptc.policy`, and allowlisted-but-unavailable warnings

### Personal fork workflow retained
- repo-local maintenance commands remain stable and documented
- the preferred personal analysis profile remains `scripts/start-pi-ptc-full-tools.sh`
- `.github/workflows/ci.yml` still runs the repo-owned verification-only CI path via `npm run verify:ci`
- release publication and git workflow decisions remain manual

## Known limitations / deferred items

This release now serves as the final bounded release-readiness gate for the current manual publish model.

Still deferred:
- automated tagging or publish flow
- automated git/PR workflow built into the repo
- bridge teardown; that still depends on Pi exposing `getToolExecutor()`
- the manual operator decision to actually publish after the documented verification gate passes

Current CI state:
- verification-only workflow present at `.github/workflows/ci.yml`
- local parity command is `npm run verify:ci`
- release publication remains manual even though verification is automated

Current dependency audit baseline:
- dependency audit: `0 critical / 2 high / 3 moderate / 1 low`
- this release documents that baseline but does not remediate it yet
